# AGENTS.md

## Stable Rules
- cocoverify2 is a stage-based, artifact-oriented cocotb verification framework.
- Preserve the current architecture unless the task explicitly asks for redesign.
- Prefer small, contract-preserving changes over broad refactors.
- Trust `docs/architecture.md`, code, tests, and fresh `tmp/` artifacts before `README.md`.
- Distinguish clearly between current implementation, future plan, and design contract.

## Phase Boundaries
- Phase 1 extracts `DUTContract`.
- Phase 2 generates `TestPlan`.
- Phase 3 generates `OracleSpec`.
- Phase 4 renders deterministic cocotb artifacts.
- Phase 5 injects execution config and runs simulations.
- Do not implement Phase 6+ behavior unless the task explicitly targets those phases.
- Do not let LLM freely generate cocotb test files.
- Preferred future LLM entry points are structured artifacts in Phase 2, Phase 3, and later Phase 7.

## Artifact Contracts
- Keep artifact schemas stable unless a change is necessary and justified.
- Preserve assumptions and ambiguities explicitly; do not erase uncertainty to make outputs look cleaner.
- Keep stage inputs and outputs typed and stage-local.
- Do not mix planning, oracle, rendering, and run logic in one place.

## Naming And Layout
- Keep stage logic under `src/cocoverify2/stages/`.
- Keep execution backends under `src/cocoverify2/execution/`.
- Keep shared typed models in `src/cocoverify2/core/models.py`.
- Use explicit artifact names like `contract.json`, `test_plan.json`, `oracle.json`, `metadata.json`, and `simulation_result.json`.
- Prefer descriptive, phase-aligned names over generic helper names.

## Makefile Contract
- Makefile is a real Phase 5 execution path, not a placeholder artifact.
- Phase 4 must render an executable Makefile shell, not a scaffold-only file.
- Phase 5 injects `SIM`, `TOPLEVEL`, `MODULE`, `VERILOG_SOURCES`, include dirs, params, defines, and run-specific env.
- Do not treat file existence as proof that a phase contract is satisfied.
- Do not mark Makefile work complete if `make` would fail immediately due to missing targets or missing runnable structure.
- If `auto` selects `make`, the rendered Makefile must already satisfy the executable-shell contract.

## Testing Expectations
- Do not rely only on happy-path or file-exists tests.
- Tests must check phase contracts, not only superficial artifact presence.
- Add regression tests for every meaningful contract bug you fix.
- For execution paths, test both runner selection and actual backend readiness.
- A test that only proves a backend was selected is insufficient if it does not prove the backend is runnable.
- Prefer hermetic tests with monkeypatching or stubs when external tools are unavailable.

## Failure Analysis
- Classify failures first: architecture issue, phase-boundary issue, artifact-contract issue, test issue, implementation bug, or environment issue.
- Do not mislabel a contract failure as an environment problem when execution already fails earlier due to incomplete artifacts.

## Current Project State
- Update this section when project reality changes.
- Current mainline is rule-based MVP through Phase 5; LLM is not yet in the main execution path.
- `TestPlan` is generated by `src/cocoverify2/stages/test_plan_generator.py`.
- `OracleSpec` is generated by `src/cocoverify2/stages/oracle_generator.py`.
- Phase 4 should remain deterministic rendering.
- Phase 4/5 Makefile contract is still an active review point and must not be treated as already solved unless artifacts and tests prove it.
